=========================================
COMPLETED FEATURES
=========================================

1. AUTHENTICATION AND SESSION
   - Login screen calls /api/v1/auth/login; JWT decoded for user ID, email, role
   - Signup screen calls /api/v1/auth/register then auto-logs in
   - Splash-based session restore - authenticated users skip the welcome screen
   - Auth token persisted in SharedPreferences; verified on every cold start
   - Expired-token detection (JWT exp claim) auto-clears session
   - Logout clears auth token and in-memory state but keeps local data caches
   - Delete account clears all local user caches and auth token

2. BUSINESS DETAILS
   - BusinessDetailsScreen prefills from saved business on every open
   - BusinessProvider creates or updates business via /api/v1/business
   - Business list cached locally per signed-in user (SharedPreferences)
  - MainShell loads businesses first on app shell init (sets ApiClient businessId)
   - Cache rehydrates immediately after logout/login - no re-entry required
   - Empty backend 404 does NOT overwrite valid local cache
   - ApiClient businessId and businessType updated whenever cache hydrates
   - Delete account clears user-scoped business cache
   - BusinessType dropdown: Cafe, Restaurant, Bakery, Catering, Food Truck, Home Kitchen, Others

3. INVENTORY ONBOARDING (InventoryDetailsScreen, post-Business Details)
   - New onboarding screen after Business Details
   - Fields: product name, type dropdown, production date, quantity, unit
   - Product types: Pastries, Beverages, Dairy, Snacks, Produce, Bakery, Meat, Spices, Frozen, Others
   - Units: KG, G, L, ML, PCS, DOZEN, OTHERS
   - Add Another queues products locally before batch submit
   - Save and Submit POSTs each product to /api/v1/inventory with businessId
  - Onboarding skip option removed (users complete this step before dashboard)
   - Completion flag and draft persisted per user in SharedPreferences
   - Draft restored on every open; completion flag skips screen on next visit
   - Default price of 1.0 NGN used (backend requires price > 0; UI captures core fields only)
   - Delete account: clears all user-scoped caches, onboarding draft, and profile image

4. INVENTORY MANAGEMENT (InventoryListScreen / AddProductScreen / ProductDetailScreen)
   - Full CRUD: add, edit, delete products via /api/v1/inventory
   - Product list cached per user+business in SharedPreferences
   - Cache hydrates before backend fetch so list is instant on re-open
   - Backend empty response does NOT overwrite existing local product cache
   - Low-stock, expiring-soon, and expired alerts computed from local inventory
   - Price validator enforces > 0 to avoid backend 422 errors
   - Product avatar displays backend image URL (NetworkImage) when available
   - Emoji + soft-color fallback via FoodVisuals when no image URL is present
   - FoodVisuals maps 60+ Nigerian/common food keywords to emoji and background color
   - imageUrl decoded from 6 server key variants: imageUrl, image_url, image, photo, photoUrl, thumbnail
   - toApiJson() payload unchanged — imageUrl never sent on create/update requests

5. DAILY SALES REPORT (DailySalesReportScreen)
   - Submits sales items via /api/v1/daily-sales with businessId
   - Sales list cached per user+business (DailySalesProvider)
   - Cache hydrates before backend fetch; network error falls back to cache
   - Add/update/delete each write-through to cache immediately
   - Products populated from live inventory for dropdown selection

6. DASHBOARD (DashboardScreen)
   - Today and yesterday revenue loaded from /api/v1/daily-sales
   - Revenue summary cached per user+business (DashboardProvider)
   - Waste risk levels and daily alerts computed from inventory state
   - Smart recommendations derived from stock levels and expiry dates
   - Profile picture saved to SharedPreferences (base64)
   - Dashboard settings (notifications, compact mode) persisted locally

7. DEMAND FORECAST (DemandForecastScreen)
   - Calls ML API at https://preppal-2mb4.onrender.com/api/predict
   - Sends positive price to avoid ML 422 errors
   - Handles mixed string/num/qualitative backend responses safely
   - Partial failure tolerant: empty forecast renders gracefully
   - Product thumbnails show NetworkImage when URL available, FoodVisuals emoji fallback otherwise
   - _resolveImageUrl checks same 6 key variants as ProductModel decoder

8. ALERTS (AlertsScreen)
   - Backend alerts via /api/v1/alerts with businessId
  - Alerts are lazy-loaded when Alerts tab/screen opens (not preloaded in MainShell init)
   - Treats 404 / empty / non-JSON responses as no-data (not error)

9. PERSISTENCE (survives logout, app close, device restart)
   - All provider caches scoped per signed-in user (userId + businessId key)
   - Cold start: all providers hydrate from cache before fetching from backend
   - Backend empty/error responses never overwrite valid local cache
   - No data re-entry required after logout, app close, or device restart
